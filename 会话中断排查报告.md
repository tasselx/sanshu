# 会话"用着用着突然断掉"排查报告

> 排查时间：2026-06-06 22:2x
> 排查对象：Cursor 3.7.12 + sanshu MCP 日志
> 结论一句话：**不是崩溃/掉线/限流/迭代上限，而是扩展宿主（Extension Host）在模型流式输出过程中被重载，导致承载会话流的 `MainThreadCursor` 被销毁、流被取消。**

## 五、根因

这一轮 agent turn 跑了约 **24 分钟、~47 次工具调用**，模型仍在流式输出长响应时，**承载会话的主窗口扩展宿主被有序重载（exit code 0 → 立即重启）**，`MainThreadCursor` 随之销毁，进行中的流被 `[canceled]`，于是会话"做到一半突然停了"。

### 手动重载 vs 自动触发（用户关心点）

所有"自动触发"路径均被日志排除，强烈指向**人为/有序重载（手动 Reload Window、切换/关闭工作区、或主动重启扩展宿主）**：

| 自动触发可能 | 是否命中 | 依据 |
|---|---|---|
| 扩展宿主卡死后自动重载 | ❌ 无 | 全日志无 `unresponsive` 事件 |
| 自动更新触发重启 | ❌ 无 | 22:08 更新检查结果 `onUpdateNotAvailable` |
| 崩溃 / 被系统杀 (OOM) | ❌ 无 | `exit code 0, signal unknown`（有序退出，非崩溃/非 SIGKILL） |

> 日志里看不到"用户点击 Reload"这一行字面记录，但既然所有自动触发都被排除、且是干净的 code-0 退出后立即在同窗口重启——这与**手动重载/窗口状态变更**的特征完全一致。旁证：22:22 紧接着打开了 sanshu 工作区，符合"在 LanShare 会话出长响应时切走/重载了窗口"的操作序列。

## 六、sanshu 项目代码核查（结论：与本次断连无关）

用户怀疑是否 sanshu 代码自身有问题。用 `sou` 定位 + Read 实证后，排除 sanshu 嫌疑：

1. **架构隔离**：sanshu MCP 是 Cursor 的独立 stdio 子进程，架构上无法重载 Cursor 的扩展宿主；而断连正是扩展宿主重载，不在其影响范围。
2. **断连时 sanshu 未活动**：最后一次 zhi 在 `22:15:56` 已正常收口，断连在 `22:21:06`（5 分钟后），期间无任何 zhi/心跳活动。
3. **心跳实现稳健**（`src/rust/mcp/tools/interaction/mcp.rs`）：每 10s 发标准 `notifications/progress` 顶住 ~30s 超时；发送失败即置 abort 停止；弹窗结束即 `handle.abort()` 回收。轻量规范，不会搞挂宿主。
4. **stdio 通道干净**（`src/rust/mcp/server.rs`）：标准 rmcp `serve(stdio())`，日志全走 `log_important!` 到 stderr/日志文件，服务端热路径无 `println!`/写 stdout 污染 JSON-RPC 流。
5. 该会话所有 sanshu 工具调用均成功（`is_error=false`）。

> 反证：sanshu 的保活设计（600s 轮询 + 10s 心跳）专为防"长等待被 30s 超时丢弃"，且本次生效（zhi 等待 48~168s 均正常返回 Done）。断连是另一套机制（扩展宿主重载），sanshu 管不到。

## 七、关键澄清：为什么 3000 万 token 能跑、500 万反而断？

这个对比恰恰证明 **断点和 token 数量无因果关系**：

- 若是"上下文/token 到上限"导致，3000 万早该断、500 万到不了上限；现在反过来 → **token 不是原因**。
- 错误是 `[canceled] MainThreadCursor disposed`（**流被取消**），不是 `context too long`（**上下文超限**）——两者是 Cursor 里完全不同的报错，后者全日志 0 命中。

真正决定断不断的是：**流式输出期间有没有发生扩展宿主重载**。

- 3000 万那条：大概率由**很多次较短 turn** 累积，每轮出字时间短，期间没赶上重载。
- 500 万这条：**单轮闷头跑 24 分钟**，把"被一次重载正好命中"的暴露时间窗拉得极长，于是被 22:21:06 那次有序重载掐断。

> 一句话：token 数只是"账单总额"；断的根因是"长时间流式输出 + 中途一次扩展宿主重载"，与 token 多少无关。

## 八、怎么解决

### 1) 根本上：别在"流式输出中"重载/切换窗口（最直接，对症本次根因）

- agent 正在出字时，**不要** Reload Window、不要切换/关闭工作区、不要重启扩展宿主，等这一轮 turn 跑完再操作。
- 长会话尤其注意：本次单轮跑了 24 分钟，中途任何一次窗口重载都会把它打断。

### 2) 缩短"暴露窗口"：让单轮 turn 别那么长（降低被打断概率）

- **长任务开 Max Mode**：上下文窗口更大，不会被迫把一轮塞到 24 分钟。
- **拆小会话**：上下文涨大就开新 chat，单轮短、token 不堆到 520 万。
- **调大 `cursor.maxIterations`**（默认 25 → 50~100），减少正常迭代中断。
- 长耗时步骤前先用 `zhi` 打招呼，避免 5 分钟工具超时叠加。

### 扩展宿主/窗口重载何时触发？

**A. 手动/用户操作（本次最可能）**：Reload Window(Cmd+R)、Restart Extension Host、打开/切换工作区文件夹、关窗再开、Glass 多工作区切换活动项目。

**B. 自动触发（部分可用 settings 压住）**：安装/启用/禁用/更新扩展、改需重启的设置或 MCP 配置、扩展宿主无响应自动重载(记 `unresponsive`)、应用自动更新重启、崩溃/OOM 自动重启。

> 本次：无 `unresponsive`、22:08 无更新、code 0 非崩溃 → 自动触发(8/9/10)全排除 → 落在 **A 手动** 或 **B6/B7 扩展/MCP 配置变更**。

- 直接回到那条 LanShare 会话，输入「继续」，agent 会带着完整历史续跑，无需重做前面的实现。
