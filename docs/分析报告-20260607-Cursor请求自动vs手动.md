## 四、归因结论

把「计费 request 时间」与 DB 里的 bubble 时间线 + 用户消息时间对齐：

| 时间 | Tokens | 判定 | 硬依据 |
|---|---|---|---|
| **11:02** | 0.32M | **手动** ✅ | composer `createdAt`=11:02:27，用户首条消息「怎么发送图片」 |
| **11:04** | 11.48M | **Cursor 自动** 🤖 | 无用户消息；首个 turn 自动续跑，连续跑到 11:24（活动热力 11:02→11:24 不断），长上下文反复重发 |
| **11:25** | 2.44M | **Cursor 自动** 🤖 | 无用户消息；**自动摘要压缩**（11:25 一分钟 171 bubble 改写，首条用户 bubble 被重打时间戳 11:25:47） |
| **11:29** | 2.05M | **手动** ✅ | 用户消息 bubble `4d5a901a`「继续」，`requestId=adf2e186…`，11:29:18 |

**一句话**：4 条里 **2 条手动（11:02、11:29）+ 2 条 Cursor 自动（11:04 长 turn 续跑、11:25 自动摘要）**。

## 五、为什么自动那条（11:04）特别贵（11.48M tokens）

- 该 request 跨越了「取证 + zhi#1（11:05，等到 11:07）+ zhi#2（11:11，等到 11:13）+ 早期实现」。
- zhi 弹窗会**阻塞并靠心跳保活**把请求拖很久；期间每次模型迭代都**重发整段上下文**
  （含读过的大量文件、base64 图片等）。
- 这正是 sanshu 自己在日志里的告警：
  > 「间隔越短=越像保活/汇报空转，**每次都重发整段上下文烧 token**」
- 所以「长 turn + 多次 zhi 往返 + 大上下文」三者叠加，导致这条自动续跑吃掉 11.48M tokens。

## 六、建议（减少「被算成新 request」与 token 浪费）

1. **减少 zhi 往返次数**：只在「多方案抉择 / 计划变更 / 任务收尾 / 危险操作」用 zhi，
   普通进度写正文即可（与现有规则一致）。本次 4 次 zhi 其实可合并为 2 次（方案确认 1 次 + 收尾 1 次）。
2. **避免超长单 turn**：任务过长时主动收口一次，让用户用一条新消息驱动下一段，
   比让 Cursor 在同一 turn 里反复自动续跑更省（自动续跑那条最贵）。
3. **客户端设置**：开启 Auto-run、调大 `cursor.maxIterations`，减少被动中断/重开请求。
4. **大附件走路径不走 base64**：本次改造已落地（附件传本地路径），可显著降低单次上下文体积，
   直接缓解 11:04 那种「base64 撑爆上下文」的情况。
