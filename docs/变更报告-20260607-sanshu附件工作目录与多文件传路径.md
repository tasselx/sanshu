# 变更报告：sanshu 附件工作目录 + 多类型文件「传路径」改造

- 日期：2026-06-07
- 范围：弹窗附件链路（前端 Vue + 后端 Rust）+ 设置页
- 目标：解决「图片以超长 base64 内联到 MCP 响应，AI 读取失败」的问题

## 一、背景与动机

改造前，用户在 zhi 弹窗中**只能粘贴图片**，图片被转为 base64 后**内联**进 MCP 响应
（`Content::image(base64, media_type)`）。当图片较大时，base64 字符串过长，导致 AI 端
解码/读取失败。

本次改造的核心思路：**所有附件落盘到本地「工作目录」，MCP 响应只传文件的绝对路径**，
由 AI 用文件读取工具按需查看，从根本上避免超长 base64。

## 二、需求确认（经 zhi 与用户对齐）

1. 新增「工作目录」存放附件，**全局共享**，可在设置中自定义。
2. 粘贴图片 + 拖入任意类型文件 → 都保存到工作目录，**支持多个**。
3. MCP 响应改为传**绝对路径**（图片也传路径，不再内联 base64）。
4. 弹窗 UI：图片显示缩略图预览，其他文件显示文件名 + 后缀 + 图标。
5. 设置里新增「附件工作目录」管理：查看列表 / 删除单个 / 一键清空 / 打开目录。
6. 命名确保不重复；Telegram 无 GUI 路径保持现状（不发附件）。
7. 自动清理：删除工作目录中超过一周（7 天）的旧文件。

## 三、整体数据流（改造后）

```
PopupInput 采集（粘贴 base64 / 拖入真实路径）
  → 后端落盘到工作目录（save_pasted_attachment / save_dropped_attachments）
  → 返回 AttachmentInfo（path/filename/kind/ext/media_type/size）
  → McpPopup.handleSubmit 组装 attachments[]（仅路径与元信息）
  → invoke('send_mcp_response')
  → Rust send_mcp_response（GUI 走 channel / MCP-CLI 走 stdout）
  → parse_structured_response：把每个附件渲染成「[图片/文件] 文件名（大小）\n 路径: <绝对路径>」文本
  → CallToolResult::success 返回给 AI（纯文本，无内联 base64）
```

## 四、后端改动（Rust）

### 1. 配置：新增工作目录字段
- `src/rust/config/settings.rs`
  - `McpConfig` 新增 `attachment_workspace_dir: Option<String>`（为空回退默认全局目录）。
  - `default_mcp_config()` 中默认 `None`。
  - 兼容性：`Option` 字段缺失时 serde 自动按 `None` 处理，旧 `config.json` 不受影响。

### 2. 新增附件模块 `src/rust/attachments/`
- `mod.rs`（核心逻辑）
  - `resolve_workspace_dir()`：读配置自定义目录，否则默认 `<config_dir>/sanshu/workspace`，并确保创建。
  - `default_workspace_dir()`：默认全局目录。
  - `save_base64()`：保存粘贴的 base64（去 data URL 前缀、解码、落盘），无后缀默认 `png`。
  - `save_dropped()`：复制拖入文件到工作目录（保留原文件）。
  - `unique_path()`：`<主体>_<毫秒时间戳>[.<后缀>]`，冲突再追加递增计数，**确保不重复**。
  - `sanitize_stem()`：清理非法字符，防目录穿越。
  - `list()` / `delete()` / `clear()`：列表（按修改时间倒序）/ 删除单个 / 清空。
  - `cleanup_expired()`：删除「修改时间超过 7 天」的文件（`MAX_AGE_SECS = 7*24*3600`），返回删除数量。
  - `read_image_data_url()`：读取图片为 data URL（**仅供弹窗预览，不发送给 AI**）。
  - `kind_for_ext()` / `media_type_for_ext()`：图片判定与常见 MIME 推断。
- `commands.rs`（Tauri 命令）
  - `save_pasted_attachment`、`save_dropped_attachments`、`list_attachments`、
    `delete_attachment`、`clear_attachments`、`get_attachment_workspace_dir`、
    `set_attachment_workspace_dir`、`open_attachment_workspace_dir`、
    `read_attachment_preview`、`select_attachment_workspace_dir`（系统目录选择对话框）。

### 3. 注册与启动清理
- `src/rust/lib.rs`：新增 `pub mod attachments;`
- `src/rust/app/builder.rs`：注册上述 10 个命令到 `invoke_handler`。
- `src/rust/app/setup.rs`：`setup_application` 启动时调用 `cleanup_expired()`，
  自动删除工作目录中超过 7 天的旧文件（best-effort，失败不影响启动）。
  弹窗多为按需启动，故每次唤起弹窗都会顺带清理。

### 4. MCP 响应类型与解析
- `src/rust/mcp/types.rs`
  - 新增 `Attachment { path, filename, kind, ext, media_type, size }`（路径型）。
  - `McpResponse` 新增 `attachments: Vec<Attachment>`（`#[serde(default)]`），
    旧 `images: Vec<ImageAttachment>` 加 `#[serde(default)]` 以兼容历史响应。
- `src/rust/mcp/handlers/response.rs`
  - `parse_structured_response`：优先处理 `attachments`，渲染为带绝对路径的文本块，
    **不再 push `Content::image`**；`images`（旧 base64）保留为兼容分支。
  - 新增 `format_byte_size()` 辅助函数。

## 五、前端改动（Vue）

### 1. 类型 `src/frontend/types/popup.d.ts`
- 新增 `Attachment` 与 `AttachmentItem`（后者多一个仅本地预览用的 `previewUrl`）。
- `McpResponse.images` → `McpResponse.attachments`。

### 2. `PopupInput.vue`
- 状态：`uploadedImages: string[]` → `attachments: AttachmentItem[]`，新增 `isDragHovering`。
- 粘贴：`addPastedImage()` 调用 `save_pasted_attachment` 落盘拿路径，复用 dataURL 做预览。

### 4. 设置页

  显示当前目录、选择目录、恢复默认、打开目录、刷新、清空全部、文件列表（含单个删除）。
- `src/frontend/components/tabs/SettingsTab.vue`：新增「附件工作目录」折叠面板并挂载组件。

## 六、关键技术点

- **拖入拿真实路径**：Tauri v2 默认 `dragDropEnabled=true`，HTML5 drop 被拦截，改由
  webview 拖拽事件（`onDragDropEvent`，payload `type:'drop'` 携带 `paths: string[]`）提供磁盘路径。
- **粘贴无路径**：剪贴板图片是二进制，前端转 base64 交后端落盘后获得路径。
- **预览不外泄**：图片预览的 data URL 仅在弹窗本地使用，**不会进入 MCP 响应**。
- **跨进程一致**：GUI 进程落盘，MCP 进程只透传响应里的绝对路径，无需重复解析目录。

## 七、兼容性

- 旧 `config.json` 缺少新字段 → 自动 `None`，回退默认目录。
- 旧格式 base64 响应（`images`）仍可解析（兼容分支保留）。
- Telegram / 图标工坊（`tu`）/ CLI 链路未改动。
