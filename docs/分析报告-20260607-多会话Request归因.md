## 三、几个典型「自动开销大户」

- **长列表设计优化**（507 msgs / 3 usr）：06-03 11:13 一分钟 **501 个 bubble 被改写** = 超大摘要压缩。
- **项目中无版权白噪音添加**（378 msgs / 7 usr）：摘要尖峰 257；7 条用户消息但 378 bubble。
- **项目理解**（92 msgs / 2 usr）：**4 次摘要事件**，反复压缩。
- **项目目录阅读 / 练习界面 UI 优化**（341 / 234 msgs，均 1 usr）：单条消息触发超长自动续跑。
- **Chat name generation instructions**（多个）：Cursor 自动建会话给对话起名，纯后台、纯额外。

## 四、和单会话结论一致的规律

1. **你手动发起的 request 很少**（全样本平均每会话 1.7 条）。
2. **大头是自动**：长 turn 自动续跑 + 上下文撑满后的自动摘要 + 后台起名会话。
3. **越长/越大的会话，自动摘要越多越贵**（501、257、171、128… 都是一次性大改写）。

## 五、对应动作（详见 `操作清单-20260607-避免被算成新Request省Token.md`）

- 任务切短、单 turn 别拖太久 → 减少自动续跑。
- 上下文水位高就**主动新开会话** → 避免自动摘要（最隐蔽的计费来源）。
- zhi 少而精、合并确认 → 缩短 turn。
- 大附件走本地路径不走 base64 → 压小上下文（本项目已落地）。

> 备注：摘要事件数 36 为**下限**——小会话即使摘要过，改写量可能 <40/分钟而未被计入；
> 另外「长 turn 自动续跑」本身也计费，但本地不存每请求 token，无法逐条计数，只能从活动时间线推断。
