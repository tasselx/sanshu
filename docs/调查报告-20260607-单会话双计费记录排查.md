# 调查报告 - 单会话出现双计费记录排查（2026-06-07）

## 一、问题背景

用户在 Cursor 用量/计费页看到 6/7 上午两条记录，但记得只开了一次会话，疑惑为何会出现两条：

| 时间（北京） | 计费类型 | 模型 | Token |
| --- | --- | --- | --- |
| 08:49 | Included | claude-4.6-opus-high-thinking | 8.126M（812.6 万） |
| 08:26 | Included | claude-4.6-opus-high-thinking | 5.058M（505.8 万） |

## 二、结论（TL;DR）

这两条记录来自**同一个会话、同一条用户消息**，不是开了两次会话。

根因：**Cursor 按 request（模型回合）计费，而不是按会话计费**。这一轮被「强制交互保活」机制拖到约 1 小时，内部发生上百次工具调用与模型往返，每次重发不断增长的上下文，于是被拆成了多条计费记录（截图里的两条只是其中一段时间）。

## 三、证据链

1. **本地 agent-transcript：8 点没有独立会话**
   最近一个会话（本轮之前）是 6/6 23:34，今天 08:26 / 08:49 没有任何独立会话记录，排除「开了两次会话」。

2. **Cursor 全局库 `state.vscdb` 锁定唯一会话**
   - 会话 `composerData:357ad096-2364-40b3-b608-7929179d94d7`，名为「项目理解」，创建于 **08:26:34**（对上第一条记录）。
   - 该会话 **用户消息（type1）只有 1 条**，AI 气泡（type2）252+ 条，**总计 380 个 bubble**。
   - 所有 bubble 时间跨度：**08:26:37 → 09:25:13**（约 1 小时）。
   - 其中一个 bubble 的 `createdAt = 2026-06-07T00:49:20Z`，换算北京时间正好是 **08:49**，精确对上第二条记录。

3. **`ai-tracking.db` 交叉验证（最精确）**
   `ai_code_hashes` 表带 `conversationId` / `requestId` / `timestamp` / `model` 字段：
   - `conversationId = 357ad096…` 下，**真正生成代码的 requestId 只有 1 个**：`a0684f29`。
   - 时间跨度 **08:51:15 → 09:24:43**，产出 **1236 行代码**，模型 `claude-opus-4-6`（即 dashboard 上的 claude-4.6-opus-high-thinking）。
   - 说明 08:26→08:49 那段是**读代码/分析阶段**（sou、Read、Grep 等工具，不产代码，故 ai-tracking 无记录）。

4. **sanshu MCP 日志解释为什么这么长**
   `MCP user-sanshu.log` 启动水印：`POPUP_POLL_WINDOW=600s` —— `zhi` 每次保活最多挂 10 分钟。叠加「强制交互」规则（未收到"完成"指令前禁止结束本轮、保活信号必须立刻再调 zhi），单轮被持续续命到近 1 小时。

## 六、为什么「一个会话」会拆成「多条记录」

Cursor 的 token 计费单位是 **request（模型回合）**，不是会话：

1. 一个超长 agentic 回合内部会发起**多次模型 API 调用**（每完成一轮工具调用就再调一次模型）。
2. 每次调用都把**不断变大的上下文重新发一遍**，所以 token 越滚越大（5.06M → 8.13M）。
3. dashboard 按 request 把这一轮拆成多条列出（08:26、08:49…）。
4. 本地库里每个 bubble 的 `tokenCount` 都是 0，**真实计费 token 只在服务端/dashboard 上可见**。

## 七、影响与建议

- 「强制交互保活 + 大上下文」会**显著放大 token 消耗**：单轮拖得越久、上下文越大，重发成本越高。
- 建议：
  - 缩短单轮任务范围，长任务尽早收尾，避免一条消息跑满 1 小时。
  - 减少无谓的 `zhi` 保活空转。
  - 必要时调低 `cursor.maxIterations`，避免单轮无限续命。
