# 操作清单：避免「同一对话被算成新 Request」+ 省 Token

> 配套分析见 `docs/分析报告-20260607-Cursor请求自动vs手动.md`。
> 本次实测：一次会话 4 条计费 request，其中 **2 条是 Cursor 自动**
> （11:04 长 turn 续跑 11.48M tokens、11:25 自动摘要），用户并没手动触发。

## 一、先认清「什么会触发新 request」

| 触发源 | 是否手动 | 是否计费新 request | 说明 |
|---|---|---|---|
| 在聊天框发新消息 | 手动 | 是 | 你能控制 |
| 点「继续 / Continue / Resume」 | 手动 | 是 | 你能控制 |
| 同一 turn 内回答 zhi 弹窗 | 半手动 | **否**（回灌同一 request） | 但会拉长 turn，间接催生自动续跑 |
| **Agent 长 turn 自动续跑** | 自动 | **是** | 迭代多/turn 太长时 Cursor 自动开新请求 |
| **上下文逼近上限 → 自动摘要压缩** | 自动 | **是** | 最隐蔽，11:25 那条就是它 |

**结论**：省 request 的关键 = ①别让单个 turn 太长 ②别让上下文涨到触发摘要 ③减少 zhi 长等待。

## 二、客户端设置（一次性配好）

1. **开启 Auto-run**：Settings → Features → Chat → `Enable auto-run mode`
   - 减少 turn 被中途打断后又要手动「继续」（手动继续也算一条 request）。
2. **调大迭代上限**：`cursor.maxIterations` 默认 25 → 调到 50~100
   - 让一个 turn 在「一条 request」里多干点活，少触发被动分段。
3. **留意上下文水位**：本次 `contextTokenLimit=300000`，用到 50%+ 就接近摘要阈值
   - 看到上下文条变满时，**主动新开会话**比让它自动摘要更省（自动摘要那条计费且贵）。

## 三、使用习惯（每次对话照做）

4. **大任务拆成多条短消息驱动**
   - 与其发一句话让 agent 连续干 20 分钟（→ 自动续跑出 11.48M tokens 那条），
     不如分阶段：你发消息 → 它做一段收口 → 你再发下一段。手动分段比自动续跑可控且通常更省。
5. **及时新开会话**
   - 一个长任务做完就开新会话，别在同一会话里继续堆下一个无关任务（上下文越滚越大→摘要）。
6. **减少 zhi 弹窗往返次数**（与项目规则一致）
   - 只在 4 类场景用 zhi：多方案抉择 / 计划变更 / 任务收尾 / 危险操作。
   - 普通进度写正文，不要把 zhi 当「每做一步汇报一次」的心跳。
   - 多个待确认点**合并成一次 zhi**（用 choices 列多项）。
   - 本次 4 次 zhi 其实可压到 2 次（方案确认 1 + 收尾 1），能显著缩短 turn 时长。
7. **别让 zhi 弹窗开着干等太久**
   - zhi 阻塞期间靠心跳保活，turn 被拉长 → 每次迭代重发整段上下文烧 token
     （sanshu 日志原话：「间隔越短=越像保活/汇报空转，每次都重发整段上下文烧 token」）。
   - 看到弹窗尽快回，或一次把要说的说全，避免反复追问。

## 四、降低单次上下文体积（治本）

8. **大附件走「本地路径」而非 base64**
   - 已在本项目落地（附件改造）：粘贴/拖入的文件保存到工作目录，只把绝对路径发给 AI。
   - 直接缓解 11:04 那种「base64 图片 + 大量文件读取撑爆上下文 → 反复重发」的 token 黑洞。
9. **少让 agent 重复读大文件**
   - 大文件用语义检索/精确 grep 定位片段，避免整文件反复进上下文。

## 五、怎么自查（复盘某条会话的 request 归因）

- **用户消息数**（手动 request 上限）：查会话 transcript 里 `role=user` 行数，或
  state DB `composerData:<id>` 的 `fullConversationHeadersOnly` 里 `type==1` 条数。
- **是否发生自动摘要**：看 state DB 里该会话 bubble 的 `createdAt` 按分钟聚合，
  某一分钟出现「几十上百个 bubble 被同时改写」≈ 摘要/压缩事件（自动、计费）。
- **zhi 往返**：`~/Library/Application Support/sanshu/log/sanshu-mcp.log` 搜 `第 N 次 zhi 调用 / 距上次`。

---
一句话总结：**手动 request 你只发了 2 条；那条 11.48M 的天价和 11:25 的摘要都是“长 turn + 大上下文”自动滚出来的。把任务切短、上下文压小、zhi 少而精，就能把自动 request 压到最低。**
